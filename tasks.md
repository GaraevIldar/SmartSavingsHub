## Команда

|  #  | Роль                       | Зона ответственности                                                                                   |
| :-: | :------------------------- | :----------------------------------------------------------------------------------------------------- |
| Б1  | **Backend / DevOps**       | Архитектура, инфраструктура (Docker, CI), .NET backend (Guild, Voice, WebSocket-слой, фоновые сервисы) |
| Б2  | **Backend / QA**           | .NET backend (Identity, Messaging), интеграционные тесты, код-ревью                                    |
| Ф1  | **Frontend-разработчик 1** | Vue.js UI - авторизация, чат, WebSocket-клиент                                                         |
| Ф2  | **Frontend-разработчик 2** | Vue.js UI -  навигация, голос (LiveKit), администрирование, статусы                                    |

> **Б1** и **Б2** оба пишут .NET-код. Разница: **Б1** владеет архитектурой решения, инфраструктурой и берет модули с инфраструктурным уклоном (WebSocket, Voice, Guild). **Б** фокусируется на бизнес-логике (Identity, Messaging) и покрытии тестами.
>
> **Важно по E2EE:** Полноценное end-to-end шифрование **не реализуется в коде MVP**. Оно остается в архитектуре и документации как ключевая фича, но в демо сообщения передаются plain text (защищены TLS). На защите показываем схему E2EE и рассказываем, как оно будет работать в следующей фазе.

## Процесс

- **Kanban** - GitHub Projects, колонки: `Backlog → In Progress → Review → Done`.
- **WIP-лимит**: не более 2 задач на человека одновременно.
- **Еженедельный синк**: 30 мин, каждый показывает работу, обсуждаем блокеры.
- **Git flow**: feature branch → Pull Request → review (минимум 1 апрув) → merge в `main`.
- Каждый этап заканчивается **работающей демо-версией**.

## Этап 1 (неделя 1–2): "Можно зайти и увидеть сервер"

**Цель:** `docker compose up` поднимает все. Пользователь может зарегистрироваться, залогиниться, создать сервер и каналы. Фронтенд отображает layout.

**Критерий приемки:** запустить → открыть браузер → зарегистрироваться → создать сервер → увидеть его в интерфейсе.

## Этап 2 (неделя 3–4): "Можно переписываться"

**Цель:** WebSocket-соединение. Отправка и получение сообщений в реальном времени (plain text). В базе - текст, в браузере - текст. Видно кто онлайн.

**Критерий приемки:** два браузера, два пользователя, один сервер → один пишет, другой мгновенно видит → в PostgreSQL - текст сообщения.
