# UC-1: Регистрация и вход через Zitadel

## TC-1.1 Успешная регистрация

**Предусловие:** пользователь не зарегистрирован.

**Шаги:**

1. На SPA нажать кнопку **«Зарегистрироваться»**.
2. Убедиться, что произошел редирект на страницу авторизации Zitadel.
3. Заполнить регистрационную форму:

   * Email;
   * Display Name;
   * Username;
   * Password;
   * Установить чекбокс согласия с Terms of Service и Privacy Policy.
4. Подтвердить регистрацию.
5. Убедиться, что произошел редирект обратно в SPA.

**Ожидаемый результат:**

* Аккаунт успешно создан в Zitadel.
* Пользователь автоматически авторизован.
* Пользователь перенаправлен на главную страницу приложения.
* В интерфейсе отображаются имя и username пользователя.

---

## TC-1.2 Регистрация с уже существующим email

**Предусловие:** пользователь с указанным email уже существует.

**Шаги:**

1. На SPA нажать кнопку **«Зарегистрироваться»**.
2. Убедиться, что произошел редирект на страницу Zitadel.
3. Заполнить форму существующим email.
4. Заполнить остальные обязательные поля.
5. Подтвердить регистрацию.

**Ожидаемый результат:**

* Zitadel отображает сообщение об ошибке о том, что пользователь уже существует.
* Новый аккаунт не создается.
* Редирект обратно в SPA не выполняется.

---

## TC-1.3 Успешный вход

**Предусловие:** пользователь зарегистрирован.

**Шаги:**

1. На SPA нажать кнопку **«Войти»**.
2. Убедиться, что произошел редирект на страницу Zitadel.
3. Ввести email.
4. Ввести корректный пароль.
5. Подтвердить вход.
6. Убедиться, что произошел редирект обратно в SPA.

**Ожидаемый результат:**

* Пользователь успешно авторизован.
* Пользователь перенаправлен на главную страницу приложения.
* В интерфейсе отображаются имя и username пользователя.

---

## TC-1.4 Вход с неверным паролем

**Предусловие:** пользователь зарегистрирован.

**Шаги:**

1. На SPA нажать кнопку **«Войти»**.
2. Дождаться перехода на страницу Zitadel.
3. Ввести корректный email.
4. Ввести неверный пароль.
5. Подтвердить вход.

**Ожидаемый результат:**

* Zitadel отображает ошибку авторизации.
* Пользователь остается на странице входа Zitadel.
* Редирект обратно в SPA не выполняется.

---

## TC-1.5 Silent Token Refresh

**Предусловие:** пользователь авторизован.

**Шаги:**

1. Выполнить вход в приложение.
2. Дождаться истечения срока действия access token.
3. Продолжить работу в приложении без обновления страницы.

**Ожидаемый результат:**

* Access token автоматически обновляется в фоновом режиме.
* Пользователь остается авторизованным.
* Редирект на страницу Zitadel не происходит.
* Текущая сессия не прерывается.

---

# UC-2: Управление сервером

## TC-2.1 Успешное создание сервера

**Предусловие:** пользователь авторизован.

**Шаги:**

1. Нажать **«Создать сервер»**.
2. Ввести название сервера.
3. Подтвердить создание.

**Ожидаемый результат:**

* Сервер успешно создан.
* Создателю сервера автоматически назначается роль **Owner**.

---

## TC-2.2 Создание канала

**Предусловие:** пользователь имеет права на создание каналов.

**Шаги:**

1. Открыть сервер.
2. Создать текстовый канал.

**Ожидаемый результат:**

* Новый канал отображается в списке каналов.

---

## TC-2.3 Генерация инвайт-ссылки

**Шаги:**

1. Открыть настройки сервера.
2. Указать TTL ссылки.
3. Указать лимит использований.
4. Нажать **«Создать инвайт»**.

**Ожидаемый результат:**

* Сгенерирована уникальная инвайт-ссылка.
* Для ссылки сохранены TTL и лимит использований.
* Ссылка доступна для копирования.

---

## TC-2.4 Переход по валидной инвайт-ссылке

**Шаги:**

1. Перейти по действующей инвайт-ссылке.

**Ожидаемый результат:**

* Пользователь успешно добавлен на сервер.
* Сервер появляется в списке серверов пользователя.

---

## TC-2.5 Использование просроченной инвайт-ссылки

**Предусловие:** срок действия ссылки истек.

**Шаги:**

1. Перейти по просроченной ссылке.

**Ожидаемый результат:**

* Отображается ошибка о том, что срок действия ссылки истек.
* Пользователь не добавляется на сервер.

---

## TC-2.6 Превышен лимит использований инвайта

**Предусловие:** лимит использований ссылки исчерпан.

**Шаги:**

1. Перейти по ссылке после достижения лимита.

**Ожидаемый результат:**

* Отображается ошибка о недействительности ссылки.
* Пользователь не добавляется на сервер.

---

# UC-3: Отправка сообщений

## TC-3.1 Отправка сообщения

**Предусловие:** пользователь находится в текстовом канале.

**Шаги:**

1. Ввести текст сообщения.
2. Нажать **«Отправить»**.

**Ожидаемый результат:**

* Сообщение сохранено в базе данных.
* Сообщение отображается у отправителя.
* Сообщение доставляется другим пользователям через WebSocket.

---

## TC-3.2 Получение сообщения другим пользователем

**Предусловие:** второй пользователь находится онлайн в том же канале.

**Ожидаемый результат:**

* Новое сообщение появляется без перезагрузки страницы.

---

## TC-3.3 Отправка пустого сообщения

**Шаги:**

1. Попытаться отправить пустую строку.

**Ожидаемый результат:**

* Сообщение не отправляется.
* Отображается ошибка валидации.

---

## TC-3.4 Потеря соединения WebSocket

**Шаги:**

1. Отключить интернет.
2. Попытаться отправить сообщение.

**Ожидаемый результат:**

* Сообщение не отправляется.
* Пользователь получает уведомление об ошибке соединения.

---

## TC-3.5 Idempotency при отправке сообщения

**Предусловие:** пользователь находится в канале.

**Шаги:**

1. Отправить сообщение с заголовком `X-Idempotency-Key`.
2. Повторно отправить тот же запрос с тем же значением `X-Idempotency-Key`.

**Ожидаемый результат:**

* В базе данных создается только одна запись.
* В канале отображается только одно сообщение.

---

## TC-3.6 Cursor Pagination истории сообщений

**Предусловие:** в канале имеется история сообщений.

**Шаги:**

1. Открыть канал.
2. Прокрутить историю сообщений вверх.

**Ожидаемый результат:**

* Старые сообщения подгружаются автоматически.
* Перезагрузка страницы не требуется.
* Уже загруженные сообщения не дублируются.

---

## TC-3.7 Member пытается удалить чужое сообщение через API

**Предусловие:** пользователь имеет роль Member.

**Шаги:**

1. Отправить API-запрос на удаление чужого сообщения.

**Ожидаемый результат:**

* Сервер возвращает HTTP 403 Forbidden.
* Сообщение не удаляется.

---

# UC-5: Модерация

## TC-5.1 Кик пользователя

**Предусловие:** действие выполняет Owner или Admin.

**Шаги:**

1. Выбрать участника сервера.
2. Нажать **«Кик»**.

**Ожидаемый результат:**

* Пользователь удаляется с сервера.
* WebSocket-соединение закрывается.
* Пользователь отключается от голосового канала.
* Пользователь может повторно присоединиться по новой или существующей инвайт-ссылке.

---

## TC-5.2 Бан пользователя

**Шаги:**

1. Owner или Admin банит пользователя.
2. Забаненный пользователь пытается перейти по действующей инвайт-ссылке.

**Ожидаемый результат:**

* Пользователь не может присоединиться к серверу.
* Сервер возвращает ошибку доступа.
* Инвайт-ссылка не позволяет обойти бан.

---

## TC-5.3 Member пытается кикнуть пользователя через API

**Предусловие:** пользователь имеет роль Member.

**Шаги:**

1. Отправить API-запрос на исключение другого участника.

**Ожидаемый результат:**

* Сервер возвращает HTTP 403 Forbidden.
* Участник остается на сервере.
* В пользовательском интерфейсе кнопка **«Кик»** отсутствует.

---

# UC-6: Онлайн-статусы

## TC-6.1 Пользователь онлайн

**Шаги:**

1. Пользователь отправляет heartbeat.

**Ожидаемый результат:**

* Статус пользователя отображается как **Online**.

---

## TC-6.2 Пользователь становится офлайн

**Шаги:**

1. Прекратить отправку heartbeat.

**Ожидаемый результат:**

* Через 30 секунд статус меняется на **Offline**.

---

## TC-6.3 Регулярный heartbeat

**Шаги:**

1. Отправлять heartbeat каждые 20 секунд.

**Ожидаемый результат:**

* Статус пользователя остается **Online**.
